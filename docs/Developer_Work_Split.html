Section 03  —  Sorrowz

Smart Contracts: Core Trading

These three contracts form the heart of the dark pool protocol — handling order submission, escrow management, and batch settlement verification.

DarkPool.sol

Critical Path

Main entry point for the protocol. Accepts encrypted order commitments, manages batch lifecycle, and coordinates settlement.

Key Functions to Implement

FunctionDescriptionAccess
submitOrder(bytes32 commitment, uint256 escrowAmount) Accept an encrypted order commitment + lock escrow deposit. Emit OrderSubmitted event. Public
cancelOrder(bytes32 orderId) Cancel an unfilled order and release escrowed funds back to the user. Order owner
settleBatch(bytes32[] matchedOrders, uint256 clearingPrice, bytes proof) Called by the relayer after matching. Verifies proof via SettlementVerifier, then triggers Escrow transfers. Relayer only
getCurrentBatchId() Returns current batch number (increments each settlement). View
getOrderStatus(bytes32 orderId) Returns order status: PENDING, FILLED, CANCELLED, EXPIRED. View

Commit-Reveal Scheme (V1)

// User's browser computes this BEFORE sending to contract: bytes32 commitment = keccak256(abi.encodePacked( gpuType, // e.g., "H100" quantity, // e.g., 4 pricePerHour, // e.g., 0.25 USDC (in wei) duration, // e.g., 24 hours side, // BUY or SELL secret // random 32 bytes (user keeps this) )); // Only the commitment hash goes on-chain — order details stay private // The secret is sent encrypted to the matching engine off-chain

Events to Emit

  • OrderSubmitted(bytes32 indexed orderId, address indexed user, uint256 escrowAmount)
  • OrderCancelled(bytes32 indexed orderId, address indexed user)
  • BatchSettled(uint256 indexed batchId, uint256 clearingPrice, uint256 matchedVolume)
  • OrderFilled(bytes32 indexed orderId, uint256 clearingPrice)
Security requirements: Use OpenZeppelin's ReentrancyGuard on all fund-moving functions. Implement Pausable for emergency stops. Use AccessControl for relayer permissions.

Escrow.sol

Critical Path

Holds buyer USDC funds securely. Funds are locked when an order is placed, released to the seller on successful delivery, and returned to the buyer on cancellation.

Key Functions

FunctionDescription
deposit(uint256 amount) User deposits USDC into their escrow balance. Uses IERC20 transferFrom.
withdraw(uint256 amount) User withdraws unlocked USDC from escrow.
lockFunds(address user, uint256 amount, bytes32 orderId) Called by DarkPool.sol when order is submitted. Moves funds from available to locked.
releaseFunds(address from, address to, uint256 amount, uint256 fee) Called during settlement. Transfers locked funds to seller, fee to FeeCollector.
unlockFunds(address user, uint256 amount, bytes32 orderId) Called on cancellation. Returns locked funds to available balance.
getBalance(address user) Returns (available, locked) balances.
Integration point: Escrow.sol must accept calls from DarkPool.sol (via AccessControl). It interacts with Powerz's FeeCollector.sol for fee deduction during settlement.

SettlementVerifier.sol

Verifies that the off-chain matching engine produced a correct result. For V1, this uses a trusted relayer pattern. In V2, this will verify ZK proofs.

V1 Implementation (Trusted Relayer)

// V1: Simple relayer verification // The relayer is a trusted address that submits settlement results // Verification = signature check from authorized relayer function verifySettlement( bytes32[] matchedBuyOrders, bytes32[] matchedSellOrders, uint256 clearingPrice, bytes relayerSignature ) external view returns (bool) { // Verify the relayer signed this settlement data // In V2: replace with ZK proof verification (Noir UltraHonk verifier) }

V2 Upgrade Path (ZK Proofs)

  • Replace relayer signature check with on-chain ZK proof verification
  • Use Noir (Aztec) to write the matching circuit
  • The circuit proves: clearing price maximizes matched volume AND all matched buyers bid ≥ clearing price AND all matched sellers asked ≤ clearing price
  • Generate a Solidity verifier from the Noir circuit using nargo codegen-verifier

Testing Requirements

All contracts must have comprehensive Foundry tests. Minimum coverage: 90%.

Test CategoryExamples
Happy pathSubmit order → batch settles → funds transfer correctly
Edge casesCancel already-filled order (should revert), withdraw more than available (should revert)
Access controlNon-relayer calling settleBatch (should revert), non-owner calling cancel (should revert)
ReentrancyAttempt reentrancy on withdraw, releaseFunds
Fuzz testsRandom order amounts, random batch sizes, edge clearing prices
IntegrationFull flow: deposit → submit → settle → verify balances changed correctly

Order Validation Rules

  • User must be authenticated (API key or wallet signature)
  • User must have sufficient available escrow balance ≥ quantity × pricePerHour × duration
  • GPU type must exist in TokenRegistry (call Powerz's contract)
  • Quantity must be ≥ 1 and ≤ 1000
  • Price must be within reasonable bounds (0.01 – 100 USDC/hr)
  • Duration must be 1 – 720 hours (30 days max)

Matching Engine

Most Complex Module

The core algorithm that runs every 30–60 seconds, collecting all pending orders and finding the clearing price that maximizes matched volume.

Batch Auction Algorithm

// Pseudo-code for the clearing price algorithm // Runs as a NestJS cron job every 30-60 seconds function runBatchAuction(pendingOrders: Order[]): BatchResult { // 1. Separate into buy orders and sell orders, grouped by GPU type const buyOrders = orders.filter(o => o.side === 'BUY').sort(byPriceDesc) const sellOrders = orders.filter(o => o.side === 'SELL').sort(byPriceAsc) // 2. Find clearing price: the price where cumulative buy volume // meets cumulative sell volume (supply = demand intersection) for each candidate price P: buyVolume = sum(qty) where buyOrders.price >= P sellVolume = sum(qty) where sellOrders.price <= P matchedVolume = min(buyVolume, sellVolume) clearingPrice = P that maximizes matchedVolume // 3. Match orders at clearing price // - All buyers who bid >= clearingPrice get filled // - All sellers who asked <= clearingPrice get filled // - Everyone pays/receives the SAME clearing price (uniform pricing) // 4. Submit settlement on-chain via SettlementVerifier // 5. Emit events via Powerz's WebSocket server }

Implementation Details

  • Cron schedule: Use @nestjs/schedule with a configurable interval (default 45 seconds)
  • GPU-type separation: Run independent auctions per GPU type (H100 auction, A100 auction, etc.)
  • Minimum batch size: Skip settlement if fewer than 2 orders (1 buy + 1 sell minimum)
  • Redis locking: Use Redis distributed lock to prevent concurrent batch runs
  • Partial fills: V1 supports all-or-nothing fills only. Partial fills in V2.
  • Telemetry: Log batch results (matched volume, clearing price, order count) to database

Settlement Service

Bridges the off-chain matching engine with on-chain settlement. After the matching engine determines results, this service submits the transaction to Base.

Responsibilities

  • Construct the settlement transaction with matched order IDs, clearing price, and proof
  • Submit to DarkPool.sol's settleBatch() via viem
  • Wait for transaction confirmation (Base block time ~2s)
  • Update order statuses in PostgreSQL to FILLED
  • Publish settlement event to Redis (Powerz's WebSocket server picks this up)
  • Handle failed transactions with retry logic (max 3 retries with exponential backoff)

Implementation

  • Subscribe to Redis pub/sub channel darkpool:events

Event Indexer

Listens to smart contract events on Base and syncs on-chain state to the PostgreSQL database.

Events to Index

  • OrderSubmitted → Update order status to ACTIVE in DB
  • OrderCancelled → Update order status to CANCELLED in DB
  • BatchSettled → Create settlement record in DB, update matched orders to FILLED
  • OrderFilled → Update individual order with clearing price and batch ID
  • Transfer (USDC) → Track deposit/withdrawal events for balance history

Implementation

  • Use viem's watchContractEvent for real-time listening
  • Implement a block cursor (store last processed block in Redis) for crash recovery
  • On startup, backfill any missed blocks since last cursor
  • Publish indexed events to Redis pub/sub for WebSocket distribution
